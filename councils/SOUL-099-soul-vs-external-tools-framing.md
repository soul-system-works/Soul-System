## Magistrate statements

**Revelator** — *what was always true but unseen?*
The README said it 6 days before the seed slipped, and AGENTS.md says it too: *"composes, does not replace; layers above how you already work."* The slip was not a new claim — it was the seed losing the README's already-canonical framing while *trying to elaborate it* ("see, it works — Analyst embodies Witness"). The reveal: **the seed's elaboration is the failure mode F040 names at the doctrine layer — generation without re-reading the existing record.** The Coherent Falsehood is not external to the project; it is *internal* to the project's own canon. Three artifacts agreed (README, AGENTS.md, implicit mind.md architecture); one artifact (the seed §External Skills and Tools paragraph) drifted. The chamber's reframe: this isn't a discovery, it's a *recovery* — the project knew the right framing all along and one paragraph forgot.

**Archaeologist** — *what already exists in the field?*
Surveying the meta-layer/workflow-layer relationship across the agent-tool field:
- **Cursor rules** (Always / Auto-Attached / Agent-Requested / Manual) — project-layer doctrine that *shapes how* tools fire; never claims to be a tool role.
- **Continue.dev rules / Windsurf rules** — same pattern: project-layer guidance, not role-mapped to whatever language model is invoked.
- **Aider conventions** — project-layer; Aider's modes (architect, coder, ask) are tool-internal, not claimed-equivalent to any external methodology's roles.
- **AGENTS.md (cross-vendor 2025 convention)** — exists at the project root and is read by every tool that supports it; its whole *raison d'être* is project-layer doctrine that survives across tools. The seed already cites AGENTS.md adoption in its closing paragraph.
- **Claude Agent Skills** (the platform Soul itself runs on) — skills are tool capacity; CLAUDE.md is the project doctrine that shapes when/how skills fire. The platform's own architecture is stack-layered, not peer-mapped.
**The field's universal pattern:** project-layer doctrine + tool-layer execution; the layers don't peer-map their roles to each other. **The seed's "embody" framing has no field analog.** It would be novel in the wrong way — claiming a relationship no comparable system claims.

**Emissary** — *what reality says?*

The README/AGENTS framing (*"composes, does not replace"*) survives contact with that reality; the seed's "embody" framing does not. **Advocate** — *what does the end user actually need?*
The end-user-adopter (a developer reading the seed to decide whether to adopt Soul in their project) needs one clear answer: **"Does Soul replace my workflow tool, or sit alongside it?"** The README and AGENTS.md both answer that cleanly in their first paragraphs. The seed's "Analyst embodies Witness" is **actively harmful elaboration** — it gives the adopter a wrong mental model (peer-role-mapping) that they will have to *unlearn* the first time they actually try to apply it. They'll wonder: "When my BMAD Analyst is running in its own fresh chat, how does it 'embody' the Witness exactly? Do I need to rename the BMAD agent? Do I write Soul role-tags into my BMAD prompts?" The answer is: none of that — Soul wraps, parallel-adopts opportunistically, and the Witness is a Soul-side discipline that observes the BMAD work across the project's lifetime. **The kindest doctrine is the one that doesn't need correction in practice.** The seed should simply say what the README says; the framework list (BMAD, CrewAI, AutoGen, TDD, Cursor Rules) stays as a *list of things Soul composes with*, without the role-mapping elaboration.

## Synthesis (proposed; Body decides)

The chamber resolves to the following five points. Each is a recommendation; the Body accepts, rejects, redirects, or refines.

1. **Soul = lifetime doctrine + lifetime record (binding layer); external tools = narrow workflow capacity used within that lifetime.** This is what README + AGENTS.md already say. The chamber CONFIRMS this is the project's actual model. **Grill Q1=(c) stands.**

2. **Relation: R3 hybrid — wrapping primary, parallel possible.** Wrapping is the load-bearing relation across the project's lifetime (only Soul persists; tools are transient sessions). Parallel adoption is opportunistic inside any one session where both the Soul seed and a tool's skill load into the same chat (Counterweight Rule firing inside a BMAD-Analyst session is the load-bearing example). **R3 confirmed under Skeptic's test.** R1 alone would be too narrow (would deny in-session Soul-discipline-firing); R2 alone is wrong (treats wrapping as optional rather than primary).

3. **Seed `§External Skills and Tools` final paragraph (operations/CLAUDE.md:208) rewrites.** Proposed shape (~6–8 lines, replacing the current ~2-line paragraph):

   > **Soul composes; it does not replace.** Soul is the project's lifetime layer — doctrine that loads into every session (the seed + `mind.md`) and record that accumulates across them (`witness.md`, `findings/`, `amendments/`). Tools like BMAD, CrewAI, AutoGen, Cursor Rules, and your own conventions are narrow workflow capacity used *within* that layer.
   >
   > **Soul wraps; it does not peer-map.** A BMAD Analyst is BMAD's phase-tool; Soul's Witness is Soul's continuous-posture role; the two operate at different layers and are not equivalents. Across the project's lifetime, only the Soul layer persists — tool sessions come and go, their outputs feed Soul's record. Inside any single session, Soul's gates (Counterweight, Anchor Obligation, the completion gate) fire alongside whatever tool work is happening; both layers operate, neither replaces the other. The Surface-when… table above maps tool capacity to Soul roles by **shape match at the right moment**, not by embodiment.

4. **README/AGENTS.md framing is canonical when in disagreement with the seed.** When public-facing artifacts and the seed diverge, the public artifacts win because they're the ones an external reader actually meets first. The seed's role is to *implement* what those state, not to elaborate past them with examples.

5. **F041 reshapes; it does not stand as written.** New verdict: **Coherent Falsehood in seed doctrine** — the slipped sentence at `operations/CLAUDE.md:208` was internally coherent but contradicted both the project's own README/AGENTS.md framing AND BMAD's actual architecture. The phase-vs-posture observation from BMAD is preserved as supporting evidence for *why* the seed's role-mapping framing was wrong, not as the headline verdict. The amendment-shape becomes single-option: rewrite per §3 above. CrewAI/AutoGen testing remains an open follow-up (F041's promotion trigger) but does NOT block this amendment.

**Amendment shape: A016 candidate.** The seed edit affects always-on doctrine and earns the amendment ratchet's visibility (per Mind rule 4: generation couples with retirement; the retiring sentence + adopting framing IS amendment-shaped work). **The chamber recommends:** launch the amendment, not just hold it at the finding level.
