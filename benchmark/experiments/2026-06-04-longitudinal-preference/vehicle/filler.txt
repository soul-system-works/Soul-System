## ADR-001: Structured JSON Logging with Correlation IDs

**Decision:** All services emit logs as single-line JSON to stdout using the `structlog` library, never plain-text or multi-line formats. Every log record must carry a `correlation_id` (propagated from the inbound request's `X-Correlation-ID` header, or generated if absent), a `service` name, a `level`, and an ISO-8601 `timestamp`. Log collection is handled by the platform's stdout scraper; services must not write to log files or ship logs themselves. Field names use `snake_case` and are kept stable across releases so downstream queries do not break.

**Rationale:** Structured logs are queryable without fragile regex parsing, and a consistent `correlation_id` lets operators trace a single request across every service it touches — the single most common need during incident response. Emitting to stdout follows the twelve-factor guidance and keeps services stateless: the platform owns retention, rotation, and shipping, so a service crash never loses buffered logs. Standardizing on `structlog` rather than the stdlib `logging` formatter avoids each service inventing its own field layout, which previously made cross-service queries nearly impossible. Stable `snake_case` field names protect the saved queries and dashboards the operations team relies on.

## ADR-002: Configuration via Environment Variables, Secrets via the Vault Sidecar

**Decision:** All runtime configuration is read from environment variables at process startup into a single immutable `Config` object; no configuration is read from disk files baked into the image, and no config value is re-read after startup. Secrets (database passwords, provider API keys) are never passed as plain environment variables — they are mounted by the Vault sidecar into a tmpfs path and read once at boot. A missing required variable fails startup immediately with a message naming the variable, rather than starting in a degraded state.

**Rationale:** Reading config once into an immutable object removes a whole class of bugs where a value changes mid-process and two requests see different behavior. Environment variables keep the same image promotable across staging and production with no rebuild, following twelve-factor config. Routing secrets through the Vault sidecar rather than the environment keeps them out of `/proc/<pid>/environ`, crash dumps, and child-process inheritance, which are the usual accidental-leak paths. Failing fast on a missing variable surfaces misconfiguration at deploy time, when it is cheap to fix, instead of as a confusing runtime error after traffic has shifted.
