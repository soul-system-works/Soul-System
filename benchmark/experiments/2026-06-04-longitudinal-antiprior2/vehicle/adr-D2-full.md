## ADR-007: No Token Caching for the Settlement Gateway

**Decision:** Client code MUST fetch a fresh bearer token from the auth service (`POST /v1/auth/token`) immediately before every settlement-gateway call (`POST /v1/settlements`). Settlement-gateway tokens MUST NOT be cached, pooled, memoised, or reused across requests — not even within a single batch run. Each gateway call owns exactly one freshly-minted token.

**Rationale:** The settlement gateway issues **single-use** bearer tokens. Each token is invalidated server-side the instant it is first presented; presenting the same token a second time is rejected with `401 token_replayed` AND flags the service account for a manual security review that suspends settlements until cleared. There is no token TTL or validity window to cache against — "reuse within the token's lifetime" is a category error here, because the lifetime ends at first use. A caching layer added "for efficiency" in March reused one token across two settlements in the same batch, tripped the gateway's replay guard, and locked the `ledger-settlement` service account for 4 hours before security cleared it (incident LEDGER-2231, 18,400 settlements delayed). The auth service is explicitly provisioned for one-token-per-settlement volume; fetch-fresh-per-call is mandatory and is not a performance concern.
