## ADR-010: Structured Logging with zerolog
**Decision:** Ledger uses `rs/zerolog` for all application logging, emitting JSON-formatted log lines to stdout. Every log entry must include at minimum: `level`, `timestamp` (RFC3339Nano), `service` ("ledger"), `env`, and a `msg` field. Contextual fields such as `account_id`, `request_id`, and `merchant_id` are attached via zerolog's `With()` chaining at handler entry points. Log level is controlled via the `LOG_LEVEL` environment variable, defaulting to `info` in production and `debug` in local development. Caller information is included only at `debug` level to reduce log volume in production.

**Rationale:** Structured JSON logs integrate directly with our Datadog log pipeline without a parsing step. Zerolog's zero-allocation design has negligible performance impact at high throughput. A consistent mandatory field set means dashboards and alerts never break due to missing context. Standardising on one library prevents mixed-format lines that complicate log-based queries across the monorepo.

===ADR===

## ADR-011: Prometheus Metrics Exposure
**Decision:** Ledger exposes a `/metrics` endpoint on a dedicated internal port (`:9090`) serving Prometheus text-format metrics via the `prometheus/client_golang` library. The endpoint is not routed through the public API gateway and is reachable only from within the cluster. All custom metrics follow the naming convention `ledger_<subsystem>_<name>_<unit>` (e.g., `ledger_db_query_duration_seconds`). Histograms use the default bucket set unless the measured operation has a known latency distribution, in which case explicit buckets are defined in the metric registration block.

**Rationale:** A separate internal port for metrics prevents accidental public exposure and removes the need for authentication on the scrape endpoint. The naming convention ensures Grafana dashboards built by the platform team can locate metrics by prefix without coordination. Restricting bucket customisation to cases with known distributions avoids bucket proliferation that inflates TSDB cardinality and scrape payload size.

===ADR===

## ADR-012: Configuration via Environment Variables
**Decision:** All runtime configuration for Ledger is provided through environment variables. The service reads and validates the full configuration set at startup using a single `config.Load()` function backed by the `kelseyhightower/envconfig` library. Required variables that are absent or malformed cause the process to exit with a non-zero status and a descriptive message before accepting any traffic. A `.env.example` file in the repository root documents every variable, its type, its default (if any), and an example value. Secrets (database passwords, signing keys) are never stored in this file; their entries are annotated with `# injected by secrets manager`.

**Rationale:** Environment-based config follows the Twelve-Factor App principle, keeping the binary environment-agnostic across local, staging, and production. Centralising validation in `config.Load()` means configuration problems surface immediately at boot rather than at first use. The `.env.example` file serves as the canonical contract between the service and its operators, reducing onboarding friction.

===ADR===

## ADR-013: Secrets Storage with HashiCorp Vault
**Decision:** All secrets required by Ledger (database credentials, HMAC signing keys, third-party API tokens) are stored in HashiCorp Vault under the `secret/ledger/<env>/` path hierarchy. The service authenticates to Vault using Kubernetes service-account-bound AppRole at pod startup and caches retrieved secrets in memory for the duration of the process lifetime. Vault secret rotation is handled via a sidecar that writes updated values to a shared in-memory volume; the main process detects changes via a `SIGHUP` handler and reloads only the secrets subsystem without a full process restart. Plaintext secrets must never appear in environment variables, ConfigMaps, or container image layers.

**Rationale:** Vault provides centralised audit trails, fine-grained ACL policies, and versioned secrets, satisfying PCI-DSS requirements for secret management. In-memory caching avoids adding Vault as a hard runtime dependency for every request. The SIGHUP-based reload pattern allows secret rotation with zero process restarts, preserving all in-flight connections.

===ADR===

## ADR-014: Per-Key Rate Limiting with Token Bucket
**Decision:** Ledger enforces per-API-key rate limits using an in-process token bucket implemented via `golang.org/x/time/rate`. Each API key is allocated a bucket sized according to its registered tier (standard: 100 req/s, premium: 1000 req/s). Bucket state is stored in a process-local `sync.Map` keyed by API key identifier; limits are therefore per-instance rather than globally coordinated. Rate limit metadata (tier, current allocation) is loaded from the database during key authentication and cached in the bucket map. Requests that exceed their bucket are rejected with HTTP 429 and a `X-RateLimit-Limit` / `X-RateLimit-Remaining` header pair.

**Rationale:** An in-process token bucket avoids a round-trip to Redis on the hot path, keeping per-request overhead minimal. Per-instance enforcement is sufficient for the current deployment topology (single-region, homogeneous pod fleet with reasonably balanced traffic). The `X-RateLimit-*` headers give callers visibility into their current allocation without requiring a separate metadata endpoint.

===ADR===

## ADR-015: Database Connection Pooling with pgxpool
**Decision:** Ledger uses `pgx/v5` with `pgxpool` for all PostgreSQL access. The pool is configured at startup with the following steady-state parameters: `MaxConns` = 20, `MinConns` = 5, `MaxConnLifetime` = 30 minutes, `MaxConnIdleTime` = 5 minutes. These parameters are exposed as environment variables so operators can tune them per deployment tier without recompiling. A single shared pool instance is wired through the application's dependency graph via constructor injection; no direct calls to `database/sql` globals are permitted outside the `internal/store` package.

**Rationale:** `pgxpool` provides native connection health checking, automatic recycling of aged connections, and a clean interface for prepared statement caching. Capping `MaxConns` at 20 keeps total connections across a 10-pod deployment under the Postgres `max_connections` ceiling with headroom for administrative sessions. Confining pool access to `internal/store` enforces a single boundary for all data-layer code, making schema and query changes easy to audit.

===ADR===

## ADR-016: Graceful Shutdown Sequence
**Decision:** On receiving SIGTERM or SIGINT, the Ledger HTTP server transitions through a defined shutdown sequence: (1) the Kubernetes readiness probe endpoint begins returning 503 to drain traffic from the load balancer, (2) the process waits a configurable `SHUTDOWN_DRAIN_SECONDS` (default: 15 s) for in-flight requests to complete, (3) the HTTP server's `Shutdown(ctx)` method is called with a timeout equal to `SHUTDOWN_DRAIN_SECONDS`, (4) the database connection pool is closed, (5) the process exits with code 0. The entire sequence is coordinated via a `context.Context` with cancellation propagated from the signal handler.

**Rationale:** A structured drain window ensures that requests already accepted by the process complete normally rather than receiving connection-reset responses during a rolling deploy. Closing the pool after the HTTP server ensures no new queries are issued while the server is winding down. Exiting with code 0 on clean shutdown distinguishes orchestrated stops from crashes in process supervision and alerting tooling.

===ADR===

## ADR-017: Health and Readiness Check Endpoints
**Decision:** Ledger exposes two probe endpoints on the internal port: `GET /healthz` (liveness) and `GET /readyz` (readiness). The liveness probe returns 200 with `{"status":"ok"}` as long as the process is running and its goroutine scheduler is responsive. The readiness probe performs a lightweight database ping (`SELECT 1`) and returns 200 only if the ping completes within 200 ms; otherwise it returns 503. During graceful shutdown, `GET /readyz` returns 503 regardless of database state. Neither endpoint requires authentication. Kubernetes probe configuration references these paths with `periodSeconds: 10` and `failureThreshold: 3`.

**Rationale:** Separating liveness and readiness prevents Kubernetes from terminating a healthy pod that is temporarily saturated. The `SELECT 1` check is the minimum necessary to confirm the database path is reachable without executing application logic. The hardcoded 200 ms timeout reflects the p99 latency budget for simple queries; slower responses indicate a pool or network condition that warrants traffic removal.

===ADR===

## ADR-018: Request Correlation IDs
**Decision:** Every inbound HTTP request to Ledger is assigned a `request_id` at the API gateway layer. Ledger's entry middleware reads the `X-Request-ID` header; if absent, the middleware generates a UUID v4 as the `request_id`. The value is stored on the request `context.Context` and propagated to all downstream log entries, outbound HTTP calls (as `X-Request-ID`), and database query comments via pgx's query-tag support. The `request_id` is echoed back to callers in the `X-Request-ID` response header.

**Rationale:** Correlation IDs allow an operator to reconstruct the full call tree for any request by filtering logs and traces on a single value. Generating a fallback UUID at the middleware layer ensures internal tooling such as health checks and integration tests always produce traceable requests without requiring the caller to inject a header. Embedding the ID in Postgres query comments makes it visible in `pg_stat_activity` and slow-query logs, closing the observability gap between the application and database layers.

===ADR===

## ADR-019: Feature Flags via LaunchDarkly
**Decision:** Ledger integrates with LaunchDarkly for runtime feature flag evaluation using the Go server-side SDK with a streaming connection to LaunchDarkly's flag delivery network. Flag evaluation is synchronous and local — the SDK maintains an in-memory flag ruleset and makes no outbound network call per evaluation. All flags are namespaced under `ledger.*`. The SDK is initialised with a 5-second startup timeout; if initialisation does not complete, the service proceeds with all flags at their defined fallback values. Flag keys and fallback values are documented alongside each usage site in `internal/flags/flags.go`.

**Rationale:** The streaming SDK keeps flag state current without polling latency, enabling near-instant flag propagation across all running instances. Local evaluation keeps flag checks entirely off the per-request network budget. Documenting fallback values in code ensures the service has a defined steady-state behaviour in every deployment environment, and makes it straightforward to reason about the surface area controlled by each flag during code review.

===ADR===

## ADR-020: Read-Data Caching with Redis
**Decision:** Ledger caches frequently read, low-churn reference data (account tier configurations, merchant profile metadata) in Redis using the `go-redis/v9` client. Cached values use the key format `ledger:<entity_type>:<id>` with TTLs defined per entity type in `internal/cache/ttl.go` (default: 60 s). The application reads from cache first; on a cache miss it reads from the primary database and populates the cache before returning. Cache entries are invalidated explicitly on write operations by calling `cache.Delete(key)` immediately after the corresponding database transaction commits. The cache is treated as a performance layer only; the database remains the authoritative source.

**Rationale:** Offloading read traffic for stable reference data reduces primary database load and improves p99 read latency. Explicit invalidation on write bounds the stale-data window to the TTL in the worst case rather than requiring readers to tolerate arbitrarily old values. Treating the cache as non-authoritative means correctness is preserved regardless of cache state, and the architecture requires no special coordination between cache and database for consistency.

===ADR===

## ADR-021: Cursor-Based Response Pagination
**Decision:** All Ledger list endpoints that may return more than 100 items use opaque cursor-based pagination. Responses include a `next_cursor` field (base64-encoded sort key + ID tuple) and a `has_more` boolean. Clients supply `?cursor=<value>` to advance to the next page. Page size is controlled by `?limit=<n>` with a maximum of 500 and a default of 50. Cursors are stateless, encode only the sort position, contain no user-identifiable data, and do not expire. Offset-based (`?page=`) pagination is not supported for any list endpoint.

**Rationale:** Cursor-based pagination produces stable results when the underlying dataset is modified between page fetches, whereas offset pagination can silently skip or duplicate rows under concurrent writes — a correctness concern for financial record listings. Opaque base64 cursors allow the internal sort key structure to change without a breaking API change. Capping the maximum page size at 500 prevents single requests from producing response payloads that stress serialisation or downstream consumer memory.

===ADR===

## ADR-022: URL-Path API Versioning
**Decision:** Ledger versions its HTTP API using a URL path prefix: `/v1/`, `/v2/`, etc. Each version is implemented as a distinct `chi.Router` mounted at its prefix in `cmd/server/routes.go`. Additive, non-breaking changes (new optional fields, new endpoints) are made within the existing version. Breaking changes (removed fields, changed semantics, new required fields) require a new version increment. The previous version is supported for a minimum of 12 months after a successor is published; the deprecation timeline is advertised via `Deprecation` and `Sunset` response headers. Version-specific handler logic lives under `internal/api/v<N>/`.

**Rationale:** URL versioning is explicit, easily routable at the load balancer level, and immediately visible in logs and traces without header parsing. A minimum 12-month overlap gives integrators a fixed migration window without emergency coordination. Separating version handler trees by directory prevents accidental cross-version coupling and allows safe deletion of a version's code once its sunset date passes.

===ADR===

## ADR-023: CORS Policy
**Decision:** Ledger's API gateway enforces a strict CORS policy. The allowed origin list is loaded from the `CORS_ALLOWED_ORIGINS` environment variable (comma-separated). Wildcard origins (`*`) are prohibited in all environments, including local development. Allowed methods are restricted to `GET`, `POST`, `DELETE`, and `PATCH`. Allowed headers are limited to `Content-Type`, `Authorization`, and `X-Request-ID`. Credentialed requests are permitted (`Access-Control-Allow-Credentials: true`). Preflight responses are cached for 600 seconds via `Access-Control-Max-Age`. CORS handling is implemented using the `rs/cors` middleware.

**Rationale:** Prohibiting wildcard origins prevents arbitrary web pages from making credentialed cross-origin requests to the API. Explicit method and header allowlists reduce the cross-origin attack surface to only what the API actively uses. Loading the origin list from an environment variable keeps the policy auditable in deployment configuration and allows additions or removals without a code change or redeployment of the application binary.

===ADR===

## ADR-024: JSON Field Naming Convention
**Decision:** All JSON request and response fields in the Ledger API use `snake_case`. Go struct fields are annotated with explicit `json:"field_name"` tags; no field is serialised without a tag. `omitempty` is used only for genuinely optional fields where absence carries semantic meaning distinct from a zero value. Boolean flags follow the pattern `is_<state>` (e.g., `is_active`). Timestamps are serialised as RFC3339 strings in UTC. Monetary amounts are serialised as integers representing the smallest currency unit (e.g., cents) alongside an explicit `currency` string field.

**Rationale:** Snake_case matches the convention of every major REST API the team integrates with, reducing context-switching for consumers. Explicit JSON tags prevent accidental field name changes from Go identifier renames propagating silently into the API contract. Serialising monetary amounts as integer cents eliminates floating-point rounding ambiguity, a hard correctness requirement for a financial ledger. RFC3339 timestamps are unambiguous and parse correctly in every major language's standard library without custom formatters.

===ADR===

## ADR-025: Database Schema Migration Tooling
**Decision:** Ledger uses `golang-migrate/migrate` with the `pgx/v5` driver for all PostgreSQL schema changes. Migrations are stored as numbered SQL files in `db/migrations/` using the `<version>_<description>.up.sql` / `.down.sql` naming convention. Migration execution is integrated into the service startup: the binary runs all pending up-migrations before the HTTP server begins accepting traffic. A `make migrate-down` target is provided for local rollback. The `schema_migrations` table managed by `golang-migrate` is the authoritative record of the applied migration version. ORM-level auto-migration is explicitly disabled.

**Rationale:** File-based SQL migrations provide a full, human-readable audit trail of every schema change and are reviewable in pull requests like any other code. Running migrations at startup guarantees the schema and binary are always in sync at deploy time without requiring a separate orchestration step. Disabling ORM auto-migration prevents unexpected DDL from executing against production databases outside the reviewed migration sequence.

===ADR===

## ADR-026: Audit Logging to Append-Only Table
**Decision:** Ledger writes a structured audit event to a dedicated `audit_log` PostgreSQL table for every state-changing operation. Each row captures: `event_id` (UUID), `actor_type` (api_key, internal_service, admin_user), `actor_id`, `event_type` (namespaced string, e.g., `account.tier_changed`), `resource_type`, `resource_id`, `before_state` (JSONB), `after_state` (JSONB), and `created_at` (timestamptz). Audit writes occur within the same database transaction as the state change; they are never written asynchronously. The `audit_log` table is append-only; no application role holds DELETE or UPDATE privileges on it.

**Rationale:** Writing audit records in the same transaction as the state change guarantees that an audit entry exists if and only if the change was committed — there is no window where state changed without a corresponding record. The append-only constraint with restricted privileges satisfies PCI-DSS requirements for tamper-evident audit trails. JSONB before/after snapshots allow full state history reconstruction without requiring joins across multiple tables.

===ADR===

## ADR-027: Blue-Green Deployment Strategy
**Decision:** Ledger is deployed using a blue-green strategy managed via Argo Rollouts. At any point exactly two identical Kubernetes deployments exist: the active ("green") set receiving 100% of traffic, and the idle ("blue") set running the prior release. A promotion advances the new release to green and retains the previous green as the new blue for a 30-minute observation window. After the window, the blue deployment is scaled to zero but its pod spec is preserved for one week. Traffic shifting is managed at the Istio VirtualService layer with 0/100 splits; no partial traffic percentages are used during steady-state operation.

**Rationale:** Blue-green deployment eliminates the mixed-version window present in rolling updates, ensuring all request-handling pods run a single binary version at any point in time. The 30-minute observation window gives on-call engineers time to review metrics dashboards before the promotion is considered complete. Preserving the blue pod spec for one week enables a fast rollback by scaling up the prior deployment without waiting for a fresh image pull or new build.

===ADR===

## ADR-028: Dependency Version Pinning
**Decision:** All Go module dependencies in Ledger are pinned to exact versions recorded in `go.sum`. Dependabot is configured to open weekly pull requests for patch-level updates and monthly pull requests for minor-level updates. Major version upgrades require a deliberate commit with a brief rationale in the commit message body. The `go.mod` file must not contain `replace` directives pointing to local file paths; a CI `go mod verify` gate enforces this. The `vendor/` directory is not committed; the module cache is populated fresh from the module proxy on each CI run.

**Rationale:** Exact version pinning ensures every CI run and production build uses the identical dependency graph, eliminating the class of bugs introduced by unreviewed upstream changes landing silently between builds. Automated Dependabot patch sweeps keep the dependency graph current with security patches without requiring manual triage. Prohibiting local `replace` directives prevents development-only stubs from reaching CI or production environments undetected.

===ADR===

## ADR-029: Timezone and UTC Policy
**Decision:** All timestamps stored in Ledger's PostgreSQL database use the `timestamptz` type and are stored in UTC. All Go `time.Time` values passed between service layers carry `time.UTC` as their location; any `time.Time` received from an external source is normalised to UTC via `.UTC()` at the service boundary. The `time.Local` variable is overridden to `time.UTC` at process startup. All timestamps in API responses are serialised as RFC3339 strings ending in `Z`. Timezone conversions for display purposes are delegated entirely to API clients.

**Rationale:** A uniform UTC policy eliminates an entire class of correctness bugs introduced by daylight-saving-time transitions and system-locale differences across deployment environments. `timestamptz` in Postgres stores values with the session timezone offset stripped, ensuring consistent retrieval regardless of the connection's `TimeZone` setting. Delegating display timezone to clients keeps the service stateless with respect to locale and removes any dependency on per-user timezone preference storage server-side.

===ADR===

## ADR-030: Pre-commit Linting with golangci-lint
**Decision:** Ledger enforces code quality checks via `golangci-lint` at two points: as a pre-commit hook (installed by `make hooks`) and as a required CI step. The active linter set is defined in `.golangci.yml` at the repository root and includes: `errcheck`, `staticcheck`, `govet`, `gofmt`, `goimports`, `gosimple`, `unused`, and `godot`. The CI step runs `golangci-lint run ./...` with `--timeout 3m` and treats any finding as a build-blocking failure. The linter version is pinned in the CI image definition and in a `tools.go` file under `internal/tools/`.

**Rationale:** Running linters as a pre-commit hook surfaces style and correctness findings before code reaches review, reducing reviewer cognitive load. Pinning the linter version prevents spurious CI failures from upstream linter releases introducing new rules mid-sprint. The selected linter set covers the most impactful classes of issues — unchecked errors, dead code, formatting inconsistency — without false-positive noise from opinionated style linters that generate contentious review comments.
