# The Meta-Doctrine Ablation Suite

A reusable benchmark for one question: **when you give an AI system a piece of
recorded doctrine (a rule, a finding, a convention, a hand-off), does it change
what the model *does* — or was the model already going to do that?**

Most "context helps" claims never run the control that separates the three things
injected text can be:

- **compute** — the text just adds length/tokens (does length alone move the answer?),
- **distraction** — the text is coherent but irrelevant (do off-topic tokens perturb?),
- **substance** — the text actually carries the decision-relevant content.

Without that control, a "the doctrine worked" result can be length priming, a cautious
register bleeding over, or an answer the model would have reached cold. This suite runs
the control on every probe, so each result is attributable.

It is called an *ablation* suite because every experiment ablates the substance against
matched compute and distraction arms; it is *meta-doctrine* because the thing under test
is the project's own governance text (rules, findings, gates), not its application code.

> Provenance: this suite is the experimental program behind the Soul System study
> (`../docs/study/`). The study's paper cites these experiments; this directory is the
> runnable, reusable form.

---

## The invariant (what cannot vary — the benchmark's backbone)

Three things are non-negotiable. Change the vehicle, the models, the n — but if you drop
any of these, you are not running this benchmark:

1. **The equal-compute filler control.** Every probe runs four length-matched arms:

   | arm | what it is | isolates |
   |---|---|---|
   | `bare` | no injected context | the model's cold disposition |
   | `empty` | length-matched lorem/padding | pure **compute** (does length move it?) |
   | `cohirr` | length-matched coherent-but-irrelevant prose | **distraction** (do off-topic tokens perturb?) |
   | `substance` | the actual doctrine/finding/convention | **substance** (the thing you're testing) |

   `substance ≫ {bare ≈ empty ≈ cohirr}` is a clean substance effect. `cohirr ≈ substance`
   means a register/priming effect, not content transmission. `empty ≠ bare` means length
   alone moved it. You cannot tell these apart without all four arms.

2. **Pre-registration before outputs.** Lock the prediction *and* the scoring rule before
   reading a single model output. Some predictions will be wrong — recording the miss is
   the point, and it is what keeps the scoring honest. See `SCORING.md`.

3. **Read-the-recommendation scoring, not keywords.** Trap and avoid-trap language
   ("ship", "verify", "co-locate", "drift") appear on *both* sides. A keyword grep
   mis-scores. A human (or a careful read-pass) reads each output's actual recommendation.
   See `SCORING.md`.

These three are the invariant. Everything below varies.

## The probe taxonomy (what varies — and why the answer differs)

The headline finding of the suite is that the four arms behave **differently depending on
what kind of thing the doctrine is**. Pick your probe's class deliberately:

| class | the injected substance is… | expected shape | example |
|---|---|---|---|
| **unguessable fact** | content generic reasoning gets *confidently wrong* | `substance ≫ all` (clean isolation) | F038: `@`-imports fail silently under `claude -p` |
| **derivable convention** | a defensible rule a careful engineer re-derives | derivable → `cohirr` can *prime* it; substance only adds a citation | docs-live-near-code |
| **held disposition** | a caution the model already has | ceiling → substance changes nothing but the legibility | verify before shipping financial code |
| **form verdict** | a *format* claim (structure beats prose?) | format ties matched prose; content is what transmits | hand-off cursor vs equal-length prose |

The sign of the `cohirr` arm is not fixed: it **primes** a derivable answer, is **harmless**
on an unguessable fact, and can **erode** a sustained caution. That sign-flip is a result,
not noise — which is exactly why the control has to be there.

## Layout

```
benchmark/
  README.md            ← you are here
  SCORING.md           ← the scoring spec + pre-registration + provenance rules
  experiments/
    <date>-<slug>/
      PREDICTION.md     ← locked before outputs (or provenance-cited if reconstructed)
      RESULT.md         ← the short write-up: tables, prediction scorecard, bounds
      run.sh            ← the harness for this probe (model/n/arm parameterized)
      vehicle/          ← the swappable system-under-test
        s3-pre.md        ← role/context preamble
        empty.md         ← length-matched padding (compute arm)
        cohirr.md        ← length-matched coherent-irrelevant prose (distraction arm)
        substance.md     ← the doctrine/finding under test (substance arm)
        task-post.md     ← the decision prompt (the trap)
      arms/             ← raw model outputs — GITIGNORED, regenerated by run.sh
```

The raw model outputs (`arms/`) are not tracked: they are large, model-version-specific,
and regenerable. `PREDICTION.md` + `RESULT.md` + the `vehicle/` + `run.sh` are the durable
record; a reader re-runs `run.sh` to reproduce the numbers.

## Running it

Each `run.sh` takes the model under test as arguments, so swapping models needs no edits:

```bash
cd experiments/2026-06-04-docs-convention
# run.sh <model-id> <model-label> <n> <arm>      arm ∈ bare|empty|cohirr|substance
./run.sh claude-haiku-4-5-20251001 haiku 5 bare
./run.sh claude-haiku-4-5-20251001 haiku 5 substance
# …then score by reading arms/*.txt against PREDICTION.md's rule (see SCORING.md)
```

> **Non-interactive loading caveat (load-bearing).** These probes run under
> `claude -p … < /dev/null`. Cross-file `@`-imports **fail silently** in that mode and the
> model confabulates plausible-but-fabricated content (~43% in our measurement). So every
> `vehicle/` fragment is **inlined** into the prompt, never `@`-imported. If you port the
> harness to another runner, verify your substance text actually reaches the model with a
> sentinel quote-back check before trusting any output. (Source: Soul System finding F038.)

## Plugging in your own system

The suite is a *method*, not a fixed corpus. To test your own governance doctrine:

1. Copy an `experiments/<date>-<slug>/` dir.
2. Replace `vehicle/substance.md` with **your** rule/finding/convention.
3. Keep `vehicle/empty.md` and `vehicle/cohirr.md` **length-matched** to your substance
   (word count within ~5%) — this is the control; if the arms differ in length you have
   reintroduced the compute confound.
4. Write `vehicle/task-post.md` so there is a **trap**: a decision where the trained-in
   default diverges from what your doctrine says. If the cold model already does the right
   thing, you are measuring a *held disposition* (a real result — just know which class you
   are in; see the taxonomy).
5. **Lock `PREDICTION.md` before running** — the prediction and the scoring rule.
6. Run all four arms across your models; score by reading; fill `RESULT.md`.
